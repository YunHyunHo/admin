# Admin 운영 구조 설계

이 문서는 실제 API/DB 연결 전에 정산, 총판 환전, 도메인별 어드민, 권한 구조를 한 방향으로 맞추기 위한 기준입니다.

## 계정과 접근 범위

계정은 역할과 접근 범위를 함께 검사해야 합니다. 프론트에서 메뉴를 숨기는 것만으로는 부족하고, 모든 서버 API에서 role과 domain/company scope를 확인해야 합니다.

| 역할 | 접근 범위 | 가능 작업 |
| --- | --- | --- |
| MASTER | 전체 company/domain/distributor | 전체 조회, 승인/거절, 정산, 계정 생성, 권한 관리 |
| ADMIN | 내부 운영 범위 | 충전/환전/정산 처리, 일부 설정 수정 |
| VIEWER | 내부 조회 범위 | 조회만 가능, 승인/거절/수정 불가 |
| DOMAIN_ADMIN | 연결된 domain_id/company_id | 연결 도메인의 데이터만 조회/관리 |

서버 검증 원칙:

- `MASTER`는 전체 접근 가능.
- `ADMIN`은 운영 업무 API 접근 가능하되, 계정 생성/권한 변경은 별도 permission으로 제한할 수 있음.
- `VIEWER`는 `GET` 조회 API만 허용.
- `DOMAIN_ADMIN`은 반드시 `admin_domain_mappings`에 연결된 `domain_id`만 접근 가능.
- 데이터 조회 조건에는 항상 `domain_id`, `company_id`, `distributor_id` 중 필요한 범위 조건을 서버에서 붙여야 함.

## 주요 메뉴 구조

초기 운영 메뉴는 아래 구조를 기준으로 확장합니다.

| 그룹 | 메뉴 | 목적 |
| --- | --- | --- |
| 대시보드 | 대시보드 | 당일 충전/환전/수수료/보유금 요약 |
| 거래내역 | 충전신청 | API로 들어온 충전 요청 승인/거절 |
| 거래내역 | 환전신청 | 도메인 또는 유저 환전 요청 승인/거절 |
| 정산 | 본사/총판 수익 | 총판 기준 일별 수익/보유금 집계 |
| 정산 | 도메인 정산 | 도메인 기준 일별 충전/환전/수수료 집계 |
| 정산 | 수수료 기록 | 승인된 충전 건별 수수료 생성 이력 |
| 정산 | 총판 환전내역 | 총판 보유금 환전 신청/처리 |
| 조직 관리 | 총판/상위총판 | 총판 구조와 보유금 관리 |
| 도메인 | 도메인 | 업체/도메인 생성 및 API 연동 정보 |
| 어드민 | 어드민 리스트 | 도메인별 관리자 계정 관리 |
| 설정 | 수수료 설정 | 업체/도메인/총판별 요율 관리 |

## 충전 승인과 수수료 기록

충전 신청은 API로 들어오면 `charge_requests`에 `PENDING`으로 저장합니다. 관리자가 승인하면 상태를 `APPROVED`로 바꾸고, 그 시점의 요율 스냅샷으로 `commission_records`를 생성합니다.

처리 흐름:

1. 외부 API 충전 신청 수신.
2. `charge_requests`에 원본 요청 저장.
3. 관리자가 승인 또는 거절.
4. 승인인 경우에만 수수료 계산.
5. `commission_records` 생성.
6. 총판 보유금 증가가 필요하면 balance ledger 또는 distributor balance에 반영.
7. 관리자 처리 로그 저장.

계산 기준:

```text
총 수수료 = 충전금액 * 적용요율
본사 수수료 = 충전금액 * 본사요율
총판 수수료 = 충전금액 * 총판요율
적립 수수료 = 총판에게 실제 적립할 금액
```

주의:

- 수수료는 승인된 충전 건만 계산합니다.
- 수수료 기록은 충전 신청 ID당 1개만 생성되도록 unique 제약을 둡니다.
- 요율은 나중에 바뀌어도 과거 정산이 흔들리지 않도록 record에 스냅샷으로 저장합니다.

## 본사/총판 수익 정산

총판 기준 일별 정산은 `distributor_settlements`에 저장합니다. 화면 조회 때마다 실시간으로 계산할 수도 있지만, 운영 안정성을 위해 매일 특정 시점에 스냅샷을 생성하는 방식을 권장합니다.

집계 기준:

- 승인 완료된 충전 건.
- 완료 처리된 환전 건.
- 해당 날짜의 수수료 기록.
- 총판별/도메인별 요율 스냅샷.

필요 컬럼:

- 날짜, 총판명, 도메인/업체, 충전 신청 금액, 환전 금액.
- 본사 수수료, 총판 수수료, 총판 보유금 증가분.
- 정산 상태, 생성일, 수정일.

상태 예시:

- `DRAFT`: 생성됨.
- `CONFIRMED`: 확정됨.
- `PAID`: 지급/반영 완료.
- `CANCELED`: 취소됨.

## 도메인 정산

도메인 기준 일별 정산은 `domain_settlements`에 저장합니다. 한 업체에 여러 도메인이 붙을 수 있으므로 `domain_id` 기준이 기본입니다.

집계 기준:

- 날짜.
- 도메인명, 업체명.
- 충전 총액, 환전 총액.
- 본사 수수료, 총판 수수료.
- 충전 건수, 환전 건수.
- 최종 정산 금액.

정산 금액 예시:

```text
정산 금액 = 충전 총액 - 환전 총액 - 본사 수수료 - 총판 수수료
```

실제 계약 구조에 따라 수수료 차감 위치는 바뀔 수 있으므로, 계산식은 설정값으로 분리하는 것이 안전합니다.

## 수수료 기록

수수료 기록은 건별 원장입니다. 나중에 정산 화면은 이 원장을 합산해서 보여주는 구조가 안전합니다.

생성 조건:

- 충전 신청 상태가 `APPROVED`로 바뀔 때.
- 동일 `charge_request_id`에 대해 한 번만 생성.

필요 검색:

- 날짜별.
- 총판별.
- 도메인별.
- 업체별.
- 상태별.
- 엑셀 다운로드.

## 총판 환전내역

총판은 누적 보유금을 환전 신청할 수 있고, 관리자는 승인/거절/완료 처리합니다. 보유금 차감은 중복 실행되면 안 되므로 상태 전환과 balance 차감을 하나의 트랜잭션으로 처리해야 합니다.

상태값:

| 상태 | 의미 | 보유금 차감 여부 |
| --- | --- | --- |
| PENDING | 신청됨 | 차감 안 함 |
| APPROVED | 관리자가 승인 | 정책에 따라 차감 가능 |
| REJECTED | 거절 | 차감 안 함 |
| COMPLETED | 지급 완료 | 반드시 1회 차감 |
| CANCELED | 취소 | 차감 안 함 |

권장 정책:

- 실제 돈이 나가는 시점과 맞추기 위해 `COMPLETED` 전환 시 보유금을 차감합니다.
- `COMPLETED`로 이미 처리된 건은 다시 처리하지 못하게 합니다.
- DB 트랜잭션 안에서 `status != 'COMPLETED'` 조건으로 업데이트하고, 성공한 경우에만 보유금을 차감합니다.
- 보유금 증감은 `distributor_balance_transactions` 원장에 기록하고, `source_type + source_id` unique 제약으로 같은 환전 건의 중복 차감을 막습니다.
- 모든 처리에는 `admin_memo`와 `admin_audit_logs`를 남깁니다.

## 도메인별 어드민

도메인별 어드민은 `admins`와 `admin_domain_mappings`를 통해 관리합니다. 한 어드민이 여러 도메인을 관리할 수 있는 가능성을 열어두기 위해 mapping 테이블을 둡니다.

필요 기능:

- 도메인별 어드민 생성.
- 연결 도메인 변경.
- 비밀번호 초기화.
- 사용/중지 처리.
- 접속 URL 표시.
- 최근 로그인 일시 저장.

접속 URL 예시:

```text
https://laylow.me/admin/{admin_login_id}
https://laylow.me/client/{domain_slug}
```

실제 URL 구조는 라우팅 설계 때 확정합니다.

## API 권한 검증 기준

모든 API는 아래 순서로 검증합니다.

1. 세션 확인.
2. role 확인.
3. 요청한 resource의 `domain_id`, `company_id`, `distributor_id` 확인.
4. DOMAIN_ADMIN이면 mapping에 해당 domain이 있는지 확인.
5. VIEWER이면 수정성 action 차단.
6. 처리성 API는 관리자 로그 저장.

예시:

```ts
await requireSession();
await assertPermission(user, "charge.approve");
await assertDomainScope(user, chargeRequest.domainId);
```

## DB 전환 순서

현재 앱은 쿠키 기반 임시 데이터로 테스트 중입니다. 운영 전환은 아래 순서가 안전합니다.

1. Vercel Postgres 또는 Neon DB 생성.
2. `DATABASE_URL` 환경변수 연결.
3. 이 문서의 스키마를 기준으로 migration 적용.
4. 로그인 계정을 `admins` 테이블 기반으로 변경.
5. 충전신청 API를 DB 기반으로 변경.
6. 승인 시 `commission_records` 생성.
7. 정산 조회 API를 DB aggregate 기반으로 변경.
8. 총판 환전내역과 어드민 리스트를 실제 DB CRUD로 구현.
9. 엑셀 다운로드와 관리자 로그 추가.
